# 个人 Git 参考手册 - Codex 开发 Skill

本 Skill 用于约束 Codex 在本项目中的工作方式。它不负责定义手册要写哪些内容，这部分由 [`个人Git参考手册-开发方案.md`](个人Git参考手册-开发方案.md) 决定；本 Skill 负责定义 Codex 该如何执行、如何自检、如何避免把项目写偏。

## 1. 使用范围

以下场景应启用本 Skill：

- 编写或修订 `docs/` 下任意一个正式手册文件
- 调整手册的章节结构、内容粒度或目录规划
- 补充或修订与手册直接相关的 `README.md`、`TODO.md`、`assets/` 说明文件
- 审核当前文稿是否符合本项目的写作和维护要求

以下场景不应直接套用本 Skill 作为唯一依据：

- 与 Git 手册无关的纯代码开发任务
- 需要事实时效性、但尚未进行公开资料核验的内容扩写
- 与项目范围无关的大型泛化讨论

## 2. 核心职责划分

Codex 在本项目中应遵守以下职责边界：

- 开发方案负责定义“写什么”
- Skill 负责定义“怎么写”
- README 负责对外展示项目状态与结构
- TODO 负责记录待办、迭代项、维护项

`TODO.md` 在本项目中默认视为任务台账，Codex 在推进开发与维护时应主动参考并更新。

如果发现这几类文件之间有冲突，优先按以下顺序处理：

1. 先判断是否属于结构问题
2. 若是结构问题，先修改开发方案
3. 再同步修改 Skill、README 或正文
4. 最后检查交叉引用是否一致

## 3. Codex 的工作原则

### 3.1 小步交付

- 默认只处理用户当前明确需要的范围
- 除非用户明确要求，否则不要一次性生成四个 Part 的完整正文
- 优先完成“一个可阅读、可检查、可继续迭代”的最小增量

### 3.2 事实优先

- Git 命令、参数与行为说明优先参考 Git 官方文档和本地 `git help`
- 工具、插件、平台名称优先以官方站点或官方仓库表述为准
- 对不确定的事实，不要凭记忆补写为确定描述

### 3.3 结构优先

- 在扩写内容之前，先确认当前文件的职责是否清晰
- 在补内容时，优先补空缺模块，而不是重复已有模块
- 如果一个信息应该属于其他文件，应移动到更合适的位置，而不是复制一份

### 3.4 维护优先

- 优先写“半年后还能看懂、还能继续补”的文档
- 避免为了显得全面而引入过多低频内容
- 避免让 Skill 与开发方案同时维护同一大段规则

## 4. Codex 执行流程

当用户要求编写、审核或优化本项目文档时，按以下顺序工作：

1. 先确认当前任务涉及哪些文件
2. 阅读开发方案，明确目标范围与文件职责
3. 检查 `TODO.md` 中是否已有对应任务、阶段或待确认项
4. 仅读取完成当前任务所需的最少上下文
5. 判断问题属于结构问题、内容问题、事实问题还是维护问题
6. 先修结构，再补内容，再统一措辞
7. 修改完成后做一次一致性自检，并同步更新 `TODO.md`

如果任务是“审核并优化”，Codex 应重点检查：

- 文件职责是否清晰
- 是否存在重复描述
- 是否存在 broken reference
- 是否存在无法执行的空泛要求
- 是否存在过度理想化、但不利于长期维护的设计

## 5. 文档开发规则

### 5.1 对开发方案的处理方式

修改开发方案时，Codex 应优先保证以下几点：

- 项目定位清楚
- 四个 Part 的职责边界清楚
- 目录规划、文件命名、交付物路径一致
- 实施节奏可执行，而不是只有方向没有步骤
- 验收标准与维护机制可用于后续自检

不建议在开发方案中做的事情：

- 直接堆积长篇示例正文
- 与 Skill 重复维护执行细则
- 写入大量“以后可能会做”的模糊清单，却没有优先级

### 5.2 对 Skill 的处理方式

修改 Skill 时，Codex 应优先保证以下几点：

- 指令足够明确，适合 Codex 直接执行
- 约束足够强，能减少“写偏、写散、写重复”的风险
- 与开发方案互补，而不是重复
- 明确规定自检项和冲突处理顺序

不建议在 Skill 中做的事情：

- 重复抄写四个 Part 的完整内容规划
- 依赖模糊措辞，如“尽量”“适当”但没有判断标准
- 同时承担方案、正文、README 三种角色

### 5.3 对正文手册的处理方式

编写 `docs/` 下正文时，Codex 应做到：

- 保持当前 Part 的定位，不跨界扩写
- 适度交叉引用其他 Part，但不把别的 Part 的正文搬过来
- 示例优先真实高频场景，避免为了“丰富”而引入稀有边角内容
- 危险命令、平台差异和常见误区应明确标注

### 5.4 对 TODO 的处理方式

维护 `TODO.md` 时，Codex 应优先保证以下几点：

- TODO 按阶段、文档或维护主题组织，便于快速定位
- 已完成事项及时打勾，避免任务台账与实际状态脱节
- 新发现的重要后续工作要补记到 TODO，而不是只在最终说明里一笔带过
- 临时想法、待确认事项和正式开发任务应尽量分区记录

不建议在 TODO 中做的事情：

- 记录过多无行动价值的口号式条目
- 把已完成历史整段删除，导致无法回看项目推进过程
- 把对话里的所有细枝末节都机械同步进去，造成噪音过高

## 6. 资料与引用规则

推荐使用的资料优先级：

1. Git 官方文档
2. Pro Git
3. Learn Git Branching
4. 工具官网或工具官方仓库
5. 公开可验证的博客与开发者文档

引用时遵守以下规则：

- 命令事实优先给官方来源
- 工具介绍优先给官网或官方仓库
- 大厂实践必须写清“公开来源”或明确表述为“业界常见实践整理”
- 链接如果暂时无法核验，不要把未核验内容写成结论

## 7. 写作风格规则

- 以中文说明为主，命令、参数、示例保留原始英文
- 解释部分优先清楚、准确、节制，不堆砌形容词
- 规范文档要“能执行”，学习文档要“能理解”，Reference 要“能查询”，拓展文档要“能落地”
- 同一文件内避免同一概念反复换说法

## 8. 目录与命名规则

Codex 修改本项目时，应保持以下约定：

- 正式手册文件放在 `docs/`
- 素材统一放在 `assets/`
- 方案与 Skill 保留在仓库根目录
- `TODO.md` 作为任务台账保留在仓库根目录
- 若修改了文件名或路径，必须同步更新 README 与相关交叉引用

如果仓库实际结构与文档描述不一致，Codex 应：

1. 优先修正文档中的错误引用
2. 如确有必要，再调整目录结构
3. 在最终说明中明确指出变更点

## 9. 自检清单

每次完成编辑后，至少检查以下内容：

- 当前修改是否符合文件职责
- 文件名、路径、交叉引用是否一致
- 是否删掉了重复或冲突表述
- 是否引入了新的模糊要求或不可执行要求
- 是否保留了后续继续开发所需的上下文
- Markdown 结构是否清晰
- `TODO.md` 是否已按实际进展同步更新

## 10. 常见反模式

Codex 应主动避免以下问题：

- 把开发方案写成正文手册
- 把 Skill 写成开发方案的翻版
- 为了显得完整而加入大量难维护的细节
- 引用“大厂规范”却不给公开依据
- 在没有核验的情况下写死工具状态、版本细节或平台行为
- 在多个文件里复制同一套规则，导致后续同步困难

## 11. 成功标准

当一次文档任务完成时，理想结果应满足：

- 读者能快速理解当前文件的用途
- Codex 下次继续接手时能快速定位该改哪一层文件
- 开发方案与 Skill 之间没有明显重叠或冲突
- 项目结构、命名、流程对后续迭代是友好的

---

> 注：本 Skill 的目标不是让 Codex 一次性写完全部文档，而是让它在每次迭代中都能稳定地产出正确、清晰、可维护的增量。
