# 个人 Git 学习与使用参考手册开发方案

## 1. 项目定位

本项目旨在构建一套面向个人开发者的 Git 参考手册，兼顾学习、查询、规范落地与效率提升四个维度。产出形式以 Markdown 文档为主，按主题拆分为多个独立文件，便于长期迭代、局部维护和开源展示。

项目的核心价值有三点：

- 帮助读者系统掌握 Git 从入门到进阶的常见用法与核心原理。
- 将个人 Git 使用习惯沉淀为一套可执行、可复用、可维护的规范。
- 汇总常见工具、技巧与自动化方案，提升真实开发场景中的工作效率。

适用场景：

- 个人学习与复盘
- 日常开发中的命令查询
- 个人项目的 Git 规范建设
- 面向求职与职场衔接的实践整理

## 2. 目标读者与边界

### 2.1 目标读者

- 希望系统学习 Git 的个人开发者
- 已会基础命令，但缺少整体知识结构的使用者
- 想为自己的项目补齐 Git 规范与工作流的人
- 希望整理一套长期维护的 Git 知识库的作者本人

### 2.2 项目边界

本项目聚焦 Git 及其直接相关的使用实践，不追求覆盖以下内容：

- 不系统讲解 GitHub Actions、CI/CD 平台等大主题，只在 Git 协作上下文中做必要说明
- 不编造任何未公开的大厂内部制度
- 不把手册写成“所有命令全部覆盖”的百科全书，而是优先覆盖高频、可落地、可维护的内容

## 3. 写作原则

### 3.1 内容原则

- 以“能用、能查、能维护”为优先，而不是单纯追求篇幅
- 优先讲清高频场景，再补充底层原理与边界情况
- 理论、示例、规范和工具彼此呼应，避免四个 Part 各写各的

### 3.2 事实原则

- Git 命令、参数与行为说明以 Git 官方文档和本机 `git help` 为准
- 引用“字节、腾讯、谷歌、苹果”等实践时，仅使用公开资料、公开博客或公开文档
- 示例中的账号、仓库地址、密钥、令牌等均使用占位符，不出现真实敏感信息

### 3.3 表述原则

- 正文以中文为主，命令、参数、提交信息示例保留英文原貌
- 能用更直白的说法说明时，不强行堆砌术语
- 需要严谨时保持严谨，适合解释时补上通俗类比

## 4. 版本基线与命令表述策略

- 目标 Git 版本基线：`Git 2.23+`
- 正文优先使用 `git switch` 与 `git restore`
- `git checkout` 作为兼容命令补充说明，不作为新手默认入口
- 涉及对象格式、行为差异或版本变化时，应显式说明适用前提

## 5. 交付物与目录规划

### 5.1 当前参考文件

当前用于驱动项目开发的核心参考文件有两份：

- `个人Git参考手册-开发方案.md`
- `个人Git参考手册-开发skill.md`

两者职责应严格区分：

- 开发方案负责定义“做什么、做到什么程度、如何拆分结构”
- Skill 负责定义“Codex 如何执行、如何校验、如何避免失控”

### 5.2 计划产出文件

后续正文文档统一放在 `docs/` 目录下：

- `docs/Git学习手册.md`
- `docs/Git Reference手册.md`
- `docs/Git个人使用规范.md`
- `docs/Git实用拓展手册.md`

辅助文件与目录：

- `README.md`：对外展示项目定位、结构与开发节奏
- `TODO.md`：记录待办、迭代项、维护事项
- `assets/`：统一存放截图、流程图、配图和临时参考素材

### 5.3 推荐目录结构


## 6. 四个 Part 的内容设计

### Part 1：Git学习手册

定位：面向学习路径的主文档，帮助读者从“知道命令”走向“理解使用场景与原理”。

目标：

- 建立完整的 Git 基础认知
- 形成从本地仓库到远程协作的操作闭环
- 为进阶章节和后续规范建立共同语境

建议模块：

- Git 简介、安装与基础配置
- 本地仓库核心操作
- 远程仓库基础交互
- 分支管理与常见协作流
- 回滚、撤销与恢复
- 冲突解决
- 标签与版本标记
- 底层对象、快照模型与分布式原理
- 常见问题排查

推荐写法：

- 每个核心知识点尽量包含“专业讲解 + 通俗解读 + 示例 + 图示 + 参考链接”
- 示例以高频、可复现、便于复制的命令为主
- 图示优先使用 Mermaid

### Part 2：Git Reference手册

定位：面向日常查询的规范型参考文档，强调准确、稳定、可检索。

目标：

- 作为命令速查表与参数说明手册
- 为 Part 1 的学习内容提供精确补充
- 为 Part 3 的规范部分提供命令层面的支撑

建议模块：

- 本地仓库命令
- 远程仓库命令
- 分支与合并命令
- 版本回滚与恢复命令
- 标签与发布相关命令
- 配置、查看、排障类命令

每个命令建议采用统一结构：

1. 命令用途
2. 语法格式
3. 常用参数
4. 示例
5. 注意事项
6. 官方文档链接

### Part 3：Git个人使用规范

定位：面向个人项目与小型协作的 Git 行为规范文档。

目标：

- 将习惯写成规则
- 将规则写成可执行步骤
- 形成适合个人长期坚持的轻量化规范

建议模块：

- 分支命名与分支职责
- 提交信息格式与提交粒度
- 单人开发流程
- 小型协作流程
- 冲突处理原则
- `.gitignore`、敏感信息、发布标签等配套规范

写作要求：

- 规范必须可执行，避免口号式表述
- 区分“硬性约束”“推荐做法”“可选优化”
- 所有引用的大厂实践都要有公开依据或清楚说明是“参考思路”

### Part 4：Git实用拓展手册

定位：面向效率提升的工具与实践文档，补足“会用 Git”到“用得顺手”的差距。

目标：

- 帮助读者选择适合自己的工具链
- 提供可直接落地的效率技巧
- 说明自动化工具与 Git 的协作边界

建议模块：

- GUI 工具
- 命令行增强工具
- Git Hooks 与自动化校验
- Git LFS 与敏感信息防护
- 高频操作技巧与问题排查技巧
- 扩展学习资源

写作要求：

- 每个工具尽量说明“是什么、解决什么问题、适合什么人、如何安装/使用”
- 涉及命令行示例时注明适用环境，如 Bash、Git Bash、PowerShell

## 7. 资料来源与引用策略

### 7.1 优先级

资料来源建议按以下优先级使用：

1. Git 官方文档：`https://git-scm.com/doc`
2. Pro Git 官方书：`https://git-scm.com/book`
3. Learn Git Branching：`https://learngitbranching.js.org/`
4. 公开可验证的云厂商或技术团队文档
5. 开源社区中高质量、长期可访问的补充资料

### 7.2 引用要求

- 命令事实优先引用官方文档
- 工具介绍优先引用工具官网或官方仓库
- 大厂规范相关内容必须标明“公开来源”或写明“业界常见实践整理”
- 若链接失效，应及时替换或在 TODO 中登记待处理项

## 8. 写作与格式规范

### 8.1 Markdown 规范

- 一级标题用于文档名或 Part 名称
- 二级标题用于一级模块
- 三级标题用于子模块
- 命令、参数、路径统一使用反引号
- 多行命令示例使用代码块，并标注语言
- 流程图优先使用 Mermaid

### 8.2 示例规范

- 示例命令优先使用最小可运行场景
- 命令前后可补充“执行前条件”和“预期结果”
- 危险命令必须标注风险，例如 `git reset --hard`
- 涉及跨平台差异时，显式说明 Windows 与类 Unix 环境的差异

### 8.3 术语规范

- 同一术语在全仓库内尽量统一写法
- 初次出现的关键术语可采用“中文 + 英文”的方式说明
- 对同一概念不要混用多种不必要的译法

## 9. 实施方案与开发节奏

### 9.1 Phase 0：项目骨架准备

目标：

- 完成 `README.md`
- 完成 `TODO.md`
- 建立 `assets/` 目录
- 确认两份参考文件职责清晰且互相引用正确

### 9.2 Phase 1：优先交付学习闭环

优先完成 Part 1 的基础使用内容，包括：

- Git 简介与安装配置
- 本地仓库基础操作
- 远程仓库基础交互
- 基础问题排查

交付标准：

- 读者能够从零完成一次本地到远程的最小闭环

### 9.3 Phase 2：补齐命令速查能力

优先完成 Part 2 的高频命令部分，包括：

- `init`、`add`、`commit`、`status`、`log`
- `clone`、`remote`、`pull`、`push`
- `branch`、`switch`、`merge`
- `reset`、`revert`、`restore`

交付标准：

- 能支持最常见的开发与排障查询场景

### 9.4 Phase 3：完善进阶与规范

依次补齐：

- Part 1 的进阶与原理部分
- Part 3 的个人使用规范
- Part 4 的效率工具与拓展内容

交付标准：

- 四个 Part 形成互相支撑的完整体系

### 9.5 Phase 4：统一打磨

统一补齐以下内容：

- Mermaid 图示
- 截图与素材整理
- 术语统一
- 链接核验
- 风险提示与交叉引用

## 10. 维护机制

### 10.1 维护触发条件

以下情况出现时应考虑更新手册：

- Git 官方文档有明显行为变化
- 计划收录的工具发生停更、改名或重大变化
- 目录结构、文件命名或仓库规范发生调整
- 某个 Part 的内容已明显落后于实际使用习惯

### 10.2 维护原则

- 结构性变更先改开发方案，再改正文或 Skill
- 正文中的事实性修订优先同步到对应 Part
- 不确定是否要扩写时，优先在 `TODO.md` 登记，而不是直接扩大范围

## 11. 验收标准

本项目的单次交付至少应满足以下条件：

- 文档结构与方案一致
- 新增内容与所在 Part 的定位一致
- 命令、参数、流程说明不自相矛盾
- 交叉引用的文件名、路径、标题保持一致
- 示例具备可复现性或足够清晰的执行前提
- 链接、术语、格式、命名保持统一

## 12. 配套关系说明

- 本文件回答“项目要写成什么样”
- `个人Git参考手册-开发skill.md` 回答“Codex 应该如何写、如何检查、如何避免写偏”
- `README.md` 面向项目展示
- `TODO.md` 面向后续迭代与维护记录
